# backend-testing

Test strategy for `apps/api`: Jest unit tests with `PrismaService` mocked
out, Supertest integration tests against a real (Fastify-backed) Nest app
instance, fixtures built from the same `@shared` zod schemas the app
validates against, and the coverage bar a layer must clear before it's done.

## Goal

Unit tests prove a service's logic in isolation (no real database); Supertest
integration tests prove a route's full request→response contract end to end
(real Nest app, real Fastify instance, either a mocked or a real disposable
test database). Every fixture is built by parsing a `@shared` schema, so a
fixture can never silently drift from the real API contract.

## Jest configuration

Nest's default `jest` block (as generated by `nest new`), unchanged:

```json
// apps/api/package.json (excerpt)
"jest": {
  "moduleFileExtensions": ["js", "json", "ts"],
  "rootDir": "src",
  "testRegex": ".*\\.spec\\.ts$",
  "transform": { "^.+\\.(t|j)s$": "ts-jest" },
  "collectCoverageFrom": ["**/*.(t|j)s", "!**/*.module.ts", "!main.ts"],
  "coverageDirectory": "../coverage",
  "testEnvironment": "node"
}
```

- `*.spec.ts` files live next to the code they test (`posts.service.spec.ts`
  beside `posts.service.ts`); e2e specs live in `apps/api/test/*.e2e-spec.ts`
  with their own `test/jest-e2e.json` (Nest's default split).
- `*.module.ts` and `main.ts` are excluded from coverage — they're wiring
  (DI registration, bootstrap), not logic; testing them would mean asserting
  "this constructor was called," which doesn't catch real bugs.

## Coverage target

**80% line coverage on services, guards, pipes, and interceptors** — the
classes that hold actual logic. Controllers can stay thinner (they're mostly
delegation to a service, covered by the e2e test that exercises the route)
and don't need the same bar. A layer isn't done (`docs/WORKFLOW.md`'s
"no advancing before tests pass" gate) if a new service method shipped with
no test at all, even if the overall percentage still clears 80%.

## Unit tests — mocking `PrismaService`

```ts
// apps/api/src/modules/posts/posts.service.spec.ts
import { Test } from "@nestjs/testing";
import { NotFoundException } from "@nestjs/common";
import { PostsService } from "./posts.service";
import { PrismaService } from "../../prisma/prisma.service";
import { buildPost } from "../../../test/fixtures/post.factory";

describe("PostsService", () => {
  let service: PostsService;
  const prisma = {
    post: {
      findMany: jest.fn(),
      findUnique: jest.fn(),
      create: jest.fn(),
    },
  };

  beforeEach(async () => {
    jest.clearAllMocks();
    const module = await Test.createTestingModule({
      providers: [PostsService, { provide: PrismaService, useValue: prisma }],
    }).compile();
    service = module.get(PostsService);
  });

  it("throws NotFoundException when the post does not exist", async () => {
    prisma.post.findUnique.mockResolvedValue(null);
    await expect(service.findOneOrThrow("missing-id")).rejects.toThrow(NotFoundException);
  });

  it("returns the post when it exists", async () => {
    const post = buildPost({ id: "post-1" });
    prisma.post.findUnique.mockResolvedValue(post);
    await expect(service.findOneOrThrow("post-1")).resolves.toEqual(post);
  });
});
```

`Test.createTestingModule({...}).overrideProvider` (or, as here, simply
supplying `useValue` directly for a provider not already in the module
array) swaps the real `PrismaService` for a plain object of `jest.fn()`s —
no real database connection in a unit test.

For a mock that stays type-safe against Prisma's actual generated client
shape (catches "you mocked a method that doesn't exist" at compile time,
which a hand-rolled `{ post: { findMany: jest.fn() } }` object won't), use
`jest-mock-extended`'s `mockDeep`:

```bash
pnpm --filter api add -D jest-mock-extended
```

```ts
import { mockDeep, type DeepMockProxy } from "jest-mock-extended";
import { PrismaClient } from "@prisma/client";

let prisma: DeepMockProxy<PrismaClient>;
beforeEach(() => {
  prisma = mockDeep<PrismaClient>();
});
// prisma.post.findUnique.mockResolvedValue(...) — fully typed, autocompletes
```

## Fixtures — built from `@shared` schemas

```ts
// apps/api/test/fixtures/post.factory.ts
import type { z } from "zod";
import { postSchema } from "@shared/contracts/post";

export function buildPost(overrides: Partial<z.infer<typeof postSchema>> = {}) {
  return postSchema.parse({
    id: "00000000-0000-0000-0000-000000000001",
    title: "Test post",
    body: "Body text",
    authorId: "00000000-0000-0000-0000-000000000002",
    createdAt: new Date().toISOString(),
    ...overrides,
  });
}
```

Parsing through `postSchema` (rather than just returning a plain object
literal typed `as Post`) means a fixture that no longer matches the real
contract fails immediately, in the fixture itself, instead of passing a test
against a shape the API doesn't actually produce anymore.

## Integration/e2e — Supertest against the Fastify-backed app

```ts
// apps/api/test/posts.e2e-spec.ts
import { Test } from "@nestjs/testing";
import { FastifyAdapter, type NestFastifyApplication } from "@nestjs/platform-fastify";
import request from "supertest";
import { AppModule } from "../src/app.module";
import { PrismaService } from "../src/prisma/prisma.service";
import { testPrismaService } from "./support/test-prisma-service"; // real client, disposable test DB/schema

describe("Posts (e2e)", () => {
  let app: NestFastifyApplication;

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({ imports: [AppModule] })
      .overrideProvider(PrismaService)
      .useValue(testPrismaService)
      .compile();

    app = moduleRef.createNestApplication<NestFastifyApplication>(new FastifyAdapter());
    await app.init();
    await app.getHttpAdapter().getInstance().ready(); // Fastify must be ready before Supertest hits it
  });

  afterAll(async () => {
    await app.close();
  });

  it("GET /v1/posts returns a paginated envelope", async () => {
    const res = await request(app.getHttpServer()).get("/v1/posts?limit=5").expect(200);

    expect(res.body).toMatchObject({
      data: expect.any(Array),
      meta: { limit: 5 },
    });
  });

  it("POST /v1/posts rejects a missing title with 422", async () => {
    const res = await request(app.getHttpServer())
      .post("/v1/posts")
      .send({ body: "no title" })
      .expect(422);

    expect(res.body.error.code).toBe("VALIDATION_ERROR");
  });
});
```

- `app.getHttpAdapter().getInstance().ready()` is required for the Fastify
  adapter specifically — unlike Express, Fastify needs an explicit `ready()`
  call before its underlying server will actually accept requests, so
  skipping it produces flaky "connection refused" failures in e2e specs.
- Prefer a **real disposable Postgres test database** (a docker-compose
  test service, migrated with `prisma migrate deploy` before the suite runs)
  over a fully mocked `PrismaService` for e2e specs — the whole point of an
  e2e test is to catch what a mock can't: real constraint violations, real
  query behavior, real transaction semantics. Reserve `PrismaService`
  mocking for unit tests (above).
- Assert the **response contract** (`api-design`'s envelope shape, status
  code, error `code`), not incidental values that will change between runs
  (timestamps, generated UUIDs) — use `expect.any(...)`/`toMatchObject` for
  those fields.

## Do

- Mock `PrismaService` for unit tests; use a real disposable test database
  for e2e specs.
- Build fixtures by parsing a `@shared` zod schema, not a hand-typed object
  literal.
- Call `app.getHttpAdapter().getInstance().ready()` before the first
  Supertest request in a Fastify e2e suite.
- Keep services/guards/pipes/interceptors at 80%+ line coverage; don't ship
  a new service method with zero coverage even if the aggregate number still
  clears the bar.

## Don't

- Don't test `*.module.ts` wiring or `main.ts` bootstrap — exclude them from
  coverage and don't write specs asserting DI registration happened.
- Don't hand-type a fixture as `Post` without parsing it through
  `postSchema` — a fixture that silently drifts from the real contract is
  worse than no fixture.
- Don't assert on volatile fields (timestamps, generated IDs) with exact
  equality — use `expect.any(...)` or `toMatchObject`.
- Don't skip `ready()` on the Fastify instance in an e2e suite and then
  chase a flaky "connection refused" as if it were a real bug.
